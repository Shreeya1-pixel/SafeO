SLIDE 3 — Why ERP systems are vulnerable
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Content:
  ERP isn’t “just internal software”

  • **Free-text everywhere** — lead notes, payment memos, comments on orders
  • **Many real users** — staff, partners, sometimes the public on website forms
  • **High value in one place** — customers, payroll, suppliers, contracts
  • **Fast changes** — custom modules and forms without a security review each time

  Attackers don’t need to “hack the server” first.
  They often test **what the app accepts** — the same place your team works every day.

Speaker Notes:

Content:
  Today’s tools often stop at the wrong layer

  • **Edge / network tools** — great for DDoS, not for “is this CRM text safe?”
  • **Generic WAF** — pattern lists, **little business context** (CRM vs payment vs HR)
  • **Logs and SIEM** — you learn **after** something was saved or exported
  • **Access rights** — control **who** can open a record, not **what** they put in it

  Missing piece: a **decision at save time** with **ERP context** and a **clear trail**.

Speaker Notes:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Content:
  Without SafeO — a story that happens in real systems

  1. A user submits a **CRM web form** with hidden junk in the “message” field (injection-style payload).
  2. Odoo **stores it** like any other lead — looks harmless in the list view.
  3. Later: a report, integration, or admin screen **touches that field** → odd errors, data exposure, or a ticket storm.
  4. Mean time to notice: think **weeks or months** — not seconds.

  **With SafeO**
  • Same submit → engine scores the text **before** create.
  • **BLOCK** → user sees an error; **nothing** persisted.
  • **WARN** → saved only if you want review paths; decision **logged** on `safeo.erp.decision`.
  • Audit trail: **who, what module, score, reason** — inside Odoo.

  Punchline: “We’re not promising magic — we’re stopping the **first bad write**.”

SLIDE 7 — How it works (simple flow)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Content:
  Seven steps — easy to say on stage

  1. User does something in Odoo (form save, POST, etc.).
  2. **ir.http** path (our module extends Odoo — **no core edits**) sees the request.
  3. Payload summarized and sent to the engine (**/erp/crm/lead**, **/erp/finance/action**, etc.).
  4. **Input checks** — injection, XSS-style patterns, risky text.
  5. **Behavior signal** — “is this normal for this user?” (where enabled).
  6. **Network hint** — optional **safe / risky** boost for demos and future feeds.
  7. Odoo gets **ALLOW / WARN / BLOCK**; **BLOCK** stops the save; events hit **metrics + dashboard**.

Speaker Notes:
  “Don’t read all seven — point at the middle: **score in the API**, **enforce in Odoo**.”
  “Judges care: **inherit**, not **fork** — we stayed inside Odoo’s extension model.”
  Optional line: “Latency target: **tens of milliseconds** for the fast path — your form still feels normal.”

Visual Suggestion:
